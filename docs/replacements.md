# Замены и доработки кода относительно диплома

Документ перечисляет все случаи, где реализация в репозитории отличается от
фрагментов кода/решений из дипломной работы. Для каждого пункта показано:
**что было** в дипломе, **на что заменено** и **почему**.

Источник: дипломная работа «Разработка информационной системы для предметной
области: детский сад», главы 2 и 3.

---

## 1. Поле `medical_notes`: `JSONB` → `JSONField`

**Что было** (диплом, раздел 2.3, описание таблицы `children_child`):
> `medical_notes (jsonb, NULL — для хранения структурированных медицинских
> сведений: аллергии, хронические заболевания, рекомендации)`

В то же время в фрагменте кода `children/models.py` (раздел 3.1) уже
используется `models.JSONField`, поэтому фактически это согласование, а не
расхождение.

**Что в репозитории** (`children/models.py`):
```python
medical_notes = models.JSONField(
    'Медицинские сведения', default=dict, blank=True,
    help_text='Например: {"allergies": ["пыльца"], "chronic": []}',
)
```

**Причина:** `models.JSONField` — кросс-СУБД-абстракция Django: в PostgreSQL она
автоматически становится `JSONB` (с индексированием и оператором `?`), а в
SQLite — `TEXT` с прозрачной сериализацией. Это нужно, чтобы проект запускался
и в PostgreSQL (как описано в дипломе), и локально на SQLite без Docker.
Функциональных потерь нет.

## 2. СУБД: только PostgreSQL → двойная конфигурация (PostgreSQL/SQLite)

**Что было** (диплом, раздел 2.2 + `docker-compose.yml`):
СУБД фиксирована — `postgres:16`, подключение через `DATABASE_URL`/переменные
окружения. Для локального запуска требуется Docker.

**Что в репозитории** (`kindergarten_project/settings.py`):
```python
if os.environ.get('POSTGRES_DB'):
    DATABASES = {'default': {'ENGINE': 'django.db.backends.postgresql', ...}}
else:
    DATABASES = {'default': {'ENGINE': 'django.db.backends.sqlite3',
                             'NAME': BASE_DIR / 'db.sqlite3'}}
```

**Причина:** Согласовано с заказчиком. SQLite нужен для быстрой проверки кода
без Docker (например, при оценке диплома). При наличии переменных окружения
PostgreSQL поднимается обычным образом, поведение приложения не меняется.

## 3. Аутентификация и роли: жёсткое сопоставление → `role_required`

**Что было** (диплом, раздел 2.3, таблица `accounts_user`):
> `role (varchar(20), NOT NULL — admin, head, teacher)`. «Для аутентификации
> используется встроенная система Django Auth с расширением через кастомную
> модель пользователя (AbstractUser)».

Конкретного кода декораторов или middleware для проверки роли в дипломе нет.

**Что в репозитории:**
- `accounts/models.py` — `User(AbstractUser)` с `role = models.CharField(choices=Role.choices)`.
- `accounts/permissions.py` — декоратор `role_required('admin', 'head', ...)`,
  применяемый к каждому представлению вне админки.
- В `attendance/views.py` и `groups/views.py` дополнительно реализована
  фильтрация «воспитатель видит только свою группу» (через `staff_members`).

**Причина:** Диплом описывает требование к ролевому доступу (раздел 2.1,
требование 6: «Подсистема аутентификации»), но не приводит реализацию.
`role_required` — стандартный для Django способ оформить такие проверки;
хранение роли в самом `User` соответствует схеме из диплома.

## 4. `attendance.views.mark_attendance` — добавлены три доработки

**Что было** (диплом, раздел 3.1):
```python
selected_date_str = request.GET.get('date') or request.POST.get('date') or str(date.today())
selected_date = date.fromisoformat(selected_date_str)
...
for child in group_children:
    is_present = request.POST.get(f'present_{child.id}') == 'on'
    ...
    AttendanceRecord.objects.update_or_create(...)
```

**Что в репозитории** (`attendance/views.py`):

1. **Валидация даты.** Дипломный код падает с `ValueError`, если в `?date=`
   передать мусор. В репозитории — функция `_parse_date(raw)` с
   `try/except ValueError → today()`.
2. **Атомарность.** Сохранение всех записей за день обёрнуто в
   `with transaction.atomic():`. В дипломе записи сохраняются по одной — при
   сбое в середине цикла останется частично сохранённая отметка.
3. **Фильтр «своя группа» для воспитателя.** Если `request.user.role == 'teacher'`,
   доступная для отметки группа сужается до групп, где этот сотрудник числится.
   Иначе — flash-сообщение и редирект.

**Причина:** Доработки обусловлены требованием безопасности (роли) и
надёжности — критерии «не более одной записи за день» и «целостность
посещаемости» из раздела 2.1 диплома.

## 5. Отчёты `reports/` — описаны в дипломе, реализованы в коде

**Что было** (диплом, раздел 2.1, требование 5):

> всех групп… отчёт по кадровому составу… отчёт по движению контингента».

В дипломе в коде реализован только `dashboard` (раздел 3.1).

- `attendance_report` — реализован: за период / по группе, с разбивкой по детям.

- `staff_report` — реализован: число по должностям, средний стаж, % высшей категории.
- `contingent_report` — реализован: зачислено/отчислено/выпущено за период.

**Причина:** Закрыть функциональное требование 5 диплома целиком.

## 6. Раздача статики — добавлен WhiteNoise

**Что было:** В дипломе (раздел 3.1, `requirements.txt`) WhiteNoise отсутствует.
Подразумевается, что статика отдаётся через nginx (в дипломе не показан).

**Что в репозитории:**
- Пакет `whitenoise==6.7.0` в `requirements.txt`.
- `whitenoise.middleware.WhiteNoiseMiddleware` в `MIDDLEWARE`.
- `STATICFILES_STORAGE = 'whitenoise.storage.CompressedStaticFilesStorage'`.

**Причина:** Чтобы приложение работало «из коробки» через `gunicorn` без nginx,
в т.ч. при первом запуске демо. Это типовой подход для Django-приложений в
Docker-режиме.

## 7. Шаблоны и навигация — реализованы

**Что было:** В дипломе (раздел 3.1) перечислены файлы шаблонов
(`templates/base.html`, `attendance/mark.html` и т. д.), но содержимого нет.

**Что в репозитории:** Полный набор шаблонов (Bootstrap 5 через CDN), общий
`base.html` с навигацией, видимостью пунктов меню по ролям. Crispy-Forms
подключён, но в текущем объёме не задействован (Admin и стандартные виджеты
покрывают все формы); пакет оставлен в зависимостях, чтобы не отклоняться
от диплома.

**Причина:** Чтобы интерфейс воспитателя/руководителя был работоспособным.

## 9. `docker-compose.yml` — убран `version: '3.8'`

**Что было** (диплом, раздел 3.1):
```yaml
version: '3.8'
services:
  ...
```

**Что в репозитории:** Атрибут `version` удалён.

**Причина:** Начиная с Docker Compose v2 атрибут `version` устарел и
выводится с предупреждением `WARN: the attribute "version" is obsolete`.
Логически конфигурация эквивалентна.

## 10. Подсчёт сотрудников с высшей категорией: `qualification__icontains` → подсчёт в Python

**Что было** (первая редакция `reports/views.py:staff_report`):
```python
high_qual = qs.filter(qualification__icontains='высш').count()
```

**Что в репозитории:**
```python
# SQLite's LIKE/LOWER не сворачивает регистр для кириллицы, поэтому считаем в Python.
high_qual = sum(
    1 for q in qs.values_list('qualification', flat=True) if 'высш' in q.lower()
)
```

**Причина:** В SQLite (используется при локальной разработке)
`LIKE` и `LOWER()` работают только с ASCII и не сворачивают регистр для
кириллицы. В результате `qualification__icontains='высш'` не находил записи
со значением `'Высшая категория'` (с заглавной «В»). На PostgreSQL та же
строка работает корректно, но проект поддерживает обе СУБД, поэтому подсчёт
вынесен в Python — `str.lower()` Python'а сворачивает кириллицу корректно
независимо от backend'а. На отчёт «Кадровый состав» (`/reports/staff/`) это
напрямую влияет: метрика «С высшей категорией» теперь показывает корректное
число. Регрессионный тест:
`reports.tests.StaffReportTest.test_high_qualification_count_handles_cyrillic_case`.

## Итог

Структура приложения, набор моделей, схема БД и ключевые представления
полностью соответствуют диплома. Замены носят либо технический характер
(совместимость со стеком, безопасность, надёжность), либо доводят описанные
в дипломе требования до работающего кода (отчёты, шаблоны, ролевой доступ).
